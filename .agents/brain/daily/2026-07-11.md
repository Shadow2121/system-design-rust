# Daily Learnings: 2026-07-11

Today we covered significant ground on both the system-design-rust KV store and the meta-tooling around it.

1. **Distributed Testing Strategy**: We learned that a "node" in a distributed system is just a process on a unique port. Testing partition tolerance and replication locally is done by spawning multiple processes and terminating them to simulate node crashes.
2. **Read Concurrency Bottlenecks**: We diagnosed a severe limitation in the `kv_store`'s `ConcurrentKvStore`. Using a single `RwLock` around both data and policy meant that `get()` requests still required a write-lock (since reading updates the eviction policy). 
3. **[[split-lock-architecture|Split-Lock Architecture]]**: To fix the read concurrency, we designed a split-lock approach: an `RwLock` for the data HashMap and a `Mutex` for the eviction policy, mirroring approaches used in production caches like Moka and Caffeine.
4. **[[arena-allocator|Arena Allocator]]**: We solved the challenge of implementing safe O(1) Doubly Linked Lists in Rust for LRU/LFU cache policies by using an Arena (Vec) and replacing memory pointers with array indices (`usize`). This bypasses borrow checker issues while maintaining lightning-fast performance.
5. **Memory Consolidation**: We designed and built an automated "Dream" system that runs via a scheduled PowerShell task to bypass permission walls. It automatically stages chat logs and uses the AI to extract decisions and learnings, effectively creating this very document.
