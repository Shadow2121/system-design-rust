# Daily Learnings: 2026-07-12

Today we built the **[[consistent-hashing|consistent_hash]]** crate — completing Crate 3 in the workspace roadmap.

1. **Consistent Hashing**: I implemented a hash ring with **virtual nodes** (vnodes). The key insight is that you don't hash each physical node once onto the ring — you hash it ~150 times to create "virtual nodes." This smooths out the distribution so no single physical node gets a disproportionate slice of the key space. Without vnodes, you'd get wildly uneven distribution.

2. **Sorted Vec + Binary Search > BTreeMap**: We chose a `Vec<(u64, N)>` kept sorted, with `binary_search_by_key` for lookups. The rationale is that the ring is **read-heavy** — lookups happen on every request, but node additions/removals are rare topology changes. A flat sorted vec has much better **cache locality** than a tree, so the constant factor on those hot-path lookups is significantly lower.

3. **Minimal Key Movement Proof**: The whole point of consistent hashing is that when you add a Nth node, only ~1/N of keys need to move. We wrote a test that adds a 6th node to a 5-node ring and verifies that only ~16.7% of 10,000 keys get reassigned. This is the foundational property that makes consistent hashing viable for distributed systems — you don't reshuffle everything like you would with `key % num_nodes`.

4. **Replica Collection**: `get_replicas(key, n)` walks the ring clockwise from the key's position, collecting **distinct physical nodes** (skipping duplicate vnodes of the same physical node). This is how DynamoDB/Riak find the N successor nodes for replication. If you ask for more replicas than nodes exist, it gracefully returns all available.
